# Portable Claude Code statusline

A self contained, cross platform (macOS and headless Linux) Claude Code
statusline. It renders across two lines:

* **Line 1 (identity)**: project name (always shown, blue, derived from the git
  repo so it stays stable no matter where you cd), an optional purple `/subdir`
  suffix when you are below the repo root, a magenta `⑂ <name>` section that
  appears only inside a linked git worktree, the git branch with a clean/dirty
  dot (green when clean, red `●N` with the changed-file count when dirty) and an
  optional `↑ahead ↓behind` versus upstream, then lines changed
  (`+added/-removed`), then the model with its reasoning effort and an
  extended-thinking dot when enabled, then session wall-clock time.
* **Line 2 (usage)**: context percentage, your Claude subscription usage (5 hour
  and weekly) with progress bars, pace markers, and reset times, then session
  cost.

Segments are separated by a gray bullet. On narrow terminals the line
collapses: below 90 columns the secondary metrics (cost, lines, duration,
ahead/behind) drop; below 65 the bars, pace markers, reset times, and the whole
weekly segment drop too. Every segment has a `SHOW_*` toggle in
`statusline-config.txt` (`RESPONSIVE=0` disables the collapse).

Claude Code renders each printed line as its own status row (see the official
docs, "Display multiple lines").

This started as the script generated by the "Claude Usage" macOS menu bar app
(HamedElfayome). It has been detached from that app so it runs anywhere,
including a GUI less VPS, without the app running or installed.

## Pieces

| File | Role |
|------|------|
| `statusline.sh` | The renderer. Claude Code pipes a JSON payload to it on stdin; it prints one status line. Reads usage numbers from a cache file. Pure bash. |
| `fetch-usage.sh` | Refreshes the usage cache by calling claude.ai. The only piece with an external dependency (see Cloudflare below). |
| `statusline-config.txt` | Feature toggles and colors (which segments show, 12 vs 24 hour clock, and so on). |
| cache: `~/.claude/.statusline-usage-cache` | Written by `fetch-usage.sh`, read by `statusline.sh`. Shared format with the app, so both can coexist. |
| credentials: `~/.config/claude-statusline/credentials` | Your session cookie and org id. Lives OUTSIDE this (public) repo. Never commit it. |

Directory, branch, model, and context come free from Claude Code's stdin
payload and need no network and no cookie. Only the usage and weekly segments
need the cache.

## The Cloudflare reality (why curl-impersonate)

Usage numbers come from `https://claude.ai/api/organizations/<org>/usage`,
authenticated with a claude.ai web session cookie. That endpoint sits behind
Cloudflare, which fingerprints the TLS handshake (JA3). A plain `curl` is
answered with a 403 "Just a moment" challenge no matter what headers you send.
macOS's Swift networking stack happens to look browser like and passes; nothing
on a headless Linux box does.

So `fetch-usage.sh` uses [curl-impersonate](https://github.com/lexiforest/curl-impersonate),
which forges a real Chrome TLS fingerprint and clears the challenge.

There is no headless way to regenerate the session cookie itself. It can only be
minted by an interactive browser login, so on a server you drop a fresh cookie
by hand when the old one expires (typically months apart).

## Install on a new machine

1. **Link it.** `install.sh` in the Dotfiles root symlinks this directory to
   `~/.claude/statusline` and points `settings.json`'s `statusLine` at
   `~/.claude/statusline/statusline.sh`. Nothing else to wire.

2. **Install curl-impersonate.** Download the release for your platform from
   https://github.com/lexiforest/curl-impersonate/releases and extract the
   `curl_chrome*` wrappers plus the `curl-impersonate` binary into
   `~/.local/bin/curl-impersonate/` (keep them together; the wrapper calls the
   binary from its own directory). `fetch-usage.sh` looks there by default, so
   no PATH change is required.

   ```bash
   mkdir -p ~/.local/bin/curl-impersonate
   # macOS arm64 example (pick the asset matching your OS/arch):
   #   curl-impersonate-v1.5.6.arm64-macos.tar.gz
   # Linux x86_64 example:
   #   curl-impersonate-v1.5.6.x86_64-linux-gnu.tar.gz
   tar xzf curl-impersonate-*.tar.gz -C ~/.local/bin/curl-impersonate
   chmod +x ~/.local/bin/curl-impersonate/*
   ```

   Also ensure `jq` is installed (`brew install jq`, `apt install jq`).

3. **Provision credentials.** Create `~/.config/claude-statusline/credentials`
   with mode 600:

   ```
   SESSION_KEY=sk-ant-sid0...your cookie...
   ORG_ID=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
   ```

   ```bash
   mkdir -p ~/.config/claude-statusline
   umask 177   # so the file is created 600
   $EDITOR ~/.config/claude-statusline/credentials
   ```

   Getting the values:
   * **SESSION_KEY**: log in at claude.ai in a browser, open DevTools, then
     Application, Cookies, `https://claude.ai`, and copy the `sessionKey`
     value (it starts with `sk-ant-sid0`).
   * **ORG_ID**: the UUID in claude.ai settings URLs, or from
     `GET https://claude.ai/api/organizations`. It rarely changes.

4. **Disable the app's built in statusline** (macOS, if you keep the app). In
   Claude Usage, open Settings, the Claude Code tab, and click Reset, then
   restart Claude Code. That stops the app from rewriting
   `~/.claude/statusline-command.sh` and from fighting this setup. The app stays
   fine as a menu bar tool; it just no longer owns the statusline.

## How refresh works

`statusline.sh` renders on every prompt. When the cache is older than the
freshness window it kicks `fetch-usage.sh` in the background (detached, single
flight locked) and renders the last known numbers immediately, so the line never
blocks on the network. The next render picks up the fresh cache. No cron, no
launchd, no menu bar app required.

Optional heavier options, if you prefer a fixed cadence over refresh on render:

* **systemd timer (Linux)**: a `.timer` firing `fetch-usage.sh` every few
  minutes. Put `SESSION_KEY` and `ORG_ID` in the unit's `Environment=` or point
  it at the credentials file.
* **launchd (macOS)**: a `LaunchAgent` plist doing the same.
* **Mac pushes to VPS**: because macOS clears Cloudflare with no special client,
  a `launchd`/cron job on a Mac can run `fetch-usage.sh` and `scp` (or `rsync`)
  the resulting cache file to the VPS's `~/.claude/.statusline-usage-cache`. The
  VPS then never needs curl-impersonate or a cookie at all. This is the most
  robust option for a long lived server.

## Config

Edit `statusline-config.txt`. Toggles are self describing (`SHOW_USAGE`,
`SHOW_WEEKLY`, `USE_24_HOUR_TIME`, `COLOR_MODE`, per element colors, and so on).
Set `SHOW_USAGE=0` to drop the cookie dependent segments entirely and get a
fully offline line (directory, branch, model, context only).

## Troubleshooting

* **Usage shows `~`**: no usable cache. Run `fetch-usage.sh` by hand and read its
  exit code (see the codes at the top of the script). Common causes: no
  credentials file, no curl-impersonate on the box, or an expired cookie.
* **Exit 5 (non 200)**: a 403 from Cloudflare. Either the cookie expired (drop a
  fresh one) or impersonation is not actually active. Do NOT set an environment
  variable named `CURL_IMPERSONATE`; the curl-impersonate binary reads that name
  as its impersonation target and a bad value silently disables impersonation.
  Use `STATUSLINE_CURL` to override the client path instead.
* **Reset time looks wrong**: it renders in the box's local timezone. On a UTC
  VPS you will see UTC. Set the machine timezone if you want local.
* **Wrong or stale numbers on macOS**: the Claude Usage app may still be writing
  the shared cache. That is harmless (same format), but if you want this script
  to be the sole writer, do the Reset in step 4.
